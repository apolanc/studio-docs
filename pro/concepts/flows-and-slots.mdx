Everything your Rasa agent does in a conversation is expressed through two primitives: **flows** and **slots**. Flows define the tasks your agent can perform and the steps required to complete them. Slots are the variables that hold information as the conversation progresses. Understanding how these two primitives work — and how CALM uses them together — is the foundation for building any Rasa assistant.

## Flows

A flow is a structured sequence of steps that guides the conversation toward completing a specific task. Each flow represents one unit of work: transfer money, reset a password, book an appointment. Keeping flows focused on a single task makes them easier to build, test, and maintain.

### How CALM selects a flow

When a user sends a message, CALM's dialogue understanding component reads the conversation and the `description` field of every flow in your project. It uses that description to decide which flow (if any) to start. This means the description is not documentation — it's a runtime input that directly affects your agent's behavior.

Write descriptions that explain **when and why** a user would trigger the flow, not just what the flow does:

```yaml
flows:
  transfer_money:
    description: >
      Help users send money to a friend or family member. Triggered when
      a user wants to make a transfer, send funds, or pay someone.
```

A vague description like `"handles transfers"` will produce worse routing than a specific one that includes the user's likely phrasing.

### Step types

A flow is made up of one or more steps. Each step has a type that determines what the agent does at that point in the conversation.

| Step type | What it does |
|---|---|
| `action` | Sends a response or runs a custom Python action |
| `collect` | Asks the user for information and stores it in a slot |
| `set_slots` | Directly assigns a value to one or more slots without asking |
| `link` | Transfers control to another flow after the current one ends |
| `call` | Calls a sub-flow and returns to the current flow when it completes |
| Conditions | Branches the flow based on current slot values or context |

`call` is particularly useful for reusable logic — authentication checks, CSAT collection, or any sub-task that multiple flows need. Think of it as a function call: the sub-flow runs, then control returns to the calling flow at the step after the `call`.

### A complete example

The following flow collects a recipient and an amount, checks whether the user has sufficient funds via a custom action, and asks for final confirmation before completing the transfer.

```yaml
flows:
  transfer_money:
    description: >
      Help users send money to a friend or family member. Triggered when
      a user wants to make a transfer, send funds, or pay someone.
    steps:
      - collect: recipient

      - collect: amount
        description: The number of US dollars to send.

      - action: action_check_sufficient_funds
        next:
          - if: not slots.has_sufficient_funds
            then:
              - action: utter_insufficient_funds
                next: END
          - else: confirm

      - collect: final_confirmation
        id: confirm
        next:
          - if: not slots.final_confirmation
            then:
              - action: utter_transfer_cancelled
                next: END
          - else: done

      - action: utter_transfer_complete
        id: done
```

A few things to notice:

- **`id` labels** let you jump to a specific step from a condition branch (`next: confirm`, `next: done`).
- **Conditions on `collect` steps** let you handle rejection inline — if the user declines, you end the flow early without needing a separate flow for the cancellation case.
- **`END`** is a built-in keyword that terminates the current flow cleanly, which triggers `pattern_completed`.

## Slots

Slots are typed variables that store information during a conversation. They persist for the duration of the session and are readable in flow conditions. You define slots in `domain.yml`.

### Slot types

| Type | Use for |
|---|---|
| `text` | Names, free-form strings |
| `float` | Numbers, amounts, percentages |
| `bool` | Yes/no confirmations |
| `categorical` | A fixed set of options |
| `list` | Multiple values |
| `any` | Untyped — use sparingly |

### Slot mappings

A mapping tells Rasa how to fill a slot. For CALM assistants, the most common mapping is `from_llm`, which instructs the dialogue understanding component to extract the value from the conversation automatically.

```yaml
slots:
  recipient:
    type: text
    mappings:
      - type: from_llm

  amount:
    type: float
    mappings:
      - type: from_llm

  final_confirmation:
    type: bool
    mappings:
      - type: from_llm
```

Other available mappings include `from_entity` (for legacy NLU pipelines) and `custom` (populated by a custom action).

When a `collect` step runs, CALM asks the user the question defined for that slot, then uses the `from_llm` mapping to extract and store whatever the user says. You can also add a `description` to a `collect` step to give the LLM extra context about what to extract — useful when the slot name alone is ambiguous.

### Reading slots in conditions

Slot values are available in flow conditions via the `slots` namespace:

```yaml
- if: slots.amount > 1000
  then:
    - action: utter_large_transfer_warning
```

You can combine conditions, check for null values, and reference nested context variables. See the [configure your agent guide](/pro/guides/configure-your-agent) for more on using conditions in flows.

## Best practices

**One flow per task.** Resist the urge to build a single large flow that handles multiple outcomes. Smaller, focused flows are easier to test, easier to read, and easier for CALM to route to correctly.

**Write descriptions for the user, not the developer.** The LLM reads flow descriptions at runtime to choose which flow to start. Write them from the perspective of what a user is trying to accomplish, including synonyms and common phrasings.

**Use `call` for shared logic.** If two or more flows need to verify a user's identity, authenticate an API session, or collect satisfaction feedback, extract that logic into a sub-flow and `call` it. This keeps each flow focused and avoids duplicating steps.

**Add descriptions to `collect` steps for ambiguous slots.** A slot named `amount` is clear enough on its own, but `date` might mean a booking date, a due date, or a birth date depending on context. A `description` on the collect step gives the LLM the disambiguation hint it needs.

**Reset slots deliberately.** Slots persist for the session. If your flow can run more than once in a session (for example, a user makes two separate transfers), make sure your flow or a follow-up action resets slot values so stale data doesn't leak into the second run.
