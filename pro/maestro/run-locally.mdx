Running your Maestro agent locally lets you verify routing logic, inspect slot values at every step, and iterate on flow definitions before deploying. The Rasa CLI provides everything you need: a training command, a live inspector, and optional voice testing.

## What the inspector shows

Before walking through the steps, it helps to know what you are working with. The Rasa inspector is a browser-based interface that shows the full state of your agent at each turn:

- **Active flow name** — which flow Maestro has currently activated
- **Slot values** — what has been collected so far in this session
- **Branching decisions** — which `if`/`else` path the dialogue manager took
- **Command logs** — the internal commands the LLM generated, such as `StartFlow` and `SetSlot`

This level of visibility makes it straightforward to diagnose routing mistakes or unexpected slot extractions.

## Run your agent

<Steps>
  <Step title="Train your model">
    From your project root, run:

    ```bash
    rasa train
    ```

    Rasa reads your `config.yml`, `data/flows.yml`, and `domain.yml`, trains the command generator, and saves a model to `models/`. Training typically takes one to two minutes for small projects.

    If training fails, check `rasa.log` in your project root for detailed error messages. Common causes include mismatched slot names between `flows.yml` and `domain.yml`, or missing `utter_ask_*` responses for `collect` steps.
  </Step>

  <Step title="Start the action server (if you have custom actions)">
    If your flows include any `action_*` steps, start the action server in a separate terminal:

    ```bash
    rasa run actions
    ```

    The action server starts on port 5055 by default. Make sure `endpoints.yml` points to it:

    ```yaml endpoints.yml
    action_endpoint:
      url: http://localhost:5055/webhook
    ```

    Skip this step if your flows only use `utter_*` responses — those are served directly by the Rasa server.
  </Step>

  <Step title="Start the Rasa server">
    Start the main Rasa server:

    ```bash
    rasa run
    ```

    For interactive testing with the browser-based inspector, use:

    ```bash
    rasa inspect
    ```

    The inspector opens automatically at `http://localhost:5005`. If you are working in a GitHub Codespace, a notification appears with a link to the forwarded port — click **Open in Browser** to access the inspector.
  </Step>

  <Step title="Test with the inspector">
    Type a message in the inspector chat panel and watch the right-hand panel update in real time. After each turn you can see:

    - Which flow is active (for example, `handle_billing`)
    - The current step within that flow
    - All slot values collected so far
    - The raw commands the command generator produced

    To test Maestro's routing, try sending messages that match different flow descriptions. For example, if you have a `handle_billing` flow and a `handle_account` flow, alternating between billing-related and account-related messages should activate the corresponding flow each time.

    Add the `--debug` flag for verbose log output directly in the terminal:

    ```bash
    rasa inspect --debug
    ```

    Debug mode prints every command, policy decision, and action invocation — useful for tracing exactly why a particular routing decision was made.
  </Step>

  <Step title="Test voice locally (optional)">
    Maestro supports voice interactions through Rasa's built-in speech integrations. To test voice locally, you need a Deepgram API key for speech recognition and synthesis.

    Set your API key as an environment variable:

    ```bash
    export DEEPGRAM_API_KEY="your-deepgram-api-key"
    ```

    Then start the inspector with voice enabled:

    ```bash
    rasa inspect --voice
    ```

    The voice inspector opens in your browser with a microphone button. Click it to speak, and your agent responds with synthesized audio. This simulates a phone call experience — the same flow logic runs; only the input and output modalities change.

    If you have filler messages configured, you will hear the acknowledgment response while tool calls execute. This is the most effective way to validate filler message timing before a voice deployment.

    Deepgram is the default speech provider. Rasa also supports other [speech integrations](https://rasa.com/docs/reference/integrations/speech-integrations/) if you prefer a different provider.
  </Step>
</Steps>

## Tips for faster iteration

- **Skip retraining for response-only changes** — editing `utter_*` responses in `domain.yml` does not require retraining. Restart the Rasa server to pick up the changes.
- **Use `rasa shell` for quick text tests** — if you do not need the visual inspector, `rasa shell` gives you a plain terminal conversation interface.
- **Watch `rasa.log`** — training errors and action server connection failures are written here. It is the first place to look when something behaves unexpectedly.
- **Reset slots mid-session** — in the inspector, click **New Conversation** to clear all slot values and start a fresh session without restarting the server.
