Deploying a Maestro agent follows the same path as any standard Rasa Pro deployment. You package your trained model, configure the runtime environment, and start the Rasa server with a tracker store and event broker. This page covers the two main deployment options — Docker Compose for smaller or development deployments, and Kubernetes for production — along with a checklist of everything you should verify before going live.

## Deployment options

Rasa Pro supports two primary deployment paths. Choose based on the scale and reliability requirements of your deployment.

### Docker Compose (development and small deployments)

Docker Compose is the fastest way to get a Maestro agent running outside your local machine. It suits development environments, internal tools, and low-traffic deployments where a single server is sufficient.

Rasa maintains an official Docker Compose template that wires together the Rasa server, action server, and a PostgreSQL tracker store:

```text
https://github.com/RasaHQ/developer-edition-docker-compose
```

From provisioning a VM to a running agent typically takes around 20 minutes. The repository includes a `docker-compose.yml` with environment variable placeholders for your license key and LLM provider credentials.

To start your deployment:

```bash
# Clone the template
git clone https://github.com/RasaHQ/developer-edition-docker-compose
cd developer-edition-docker-compose

# Set required environment variables
export RASA_LICENSE="your-rasa-pro-license-key"
export OPENAI_API_KEY="your-openai-api-key"  # or your preferred LLM provider

# Start all services
docker compose up -d
```

### Kubernetes (production)

For production traffic — multiple users, high availability requirements, or integration with enterprise infrastructure — deploy Rasa Pro on Kubernetes. This requires a commercial Rasa license.

A full production Kubernetes deployment includes:

- Multiple Rasa server pods behind a load balancer
- A dedicated action server deployment
- PostgreSQL for the tracker store
- An event broker (Kafka or RabbitMQ) for analytics
- S3-compatible object storage for model files
- Kubernetes secrets for license keys and API credentials
- Optional: Rasa Studio connection for the no-code authoring interface

Rasa publishes official Kubernetes deployment guides for AWS, GCP, and Azure. Start with the Kubernetes deployment documentation:

```text
https://rasa.com/docs/pro/deploy/deploy-kubernetes/
```

<Note>
  For on-premises and air-gapped deployments — where no traffic leaves your network and no external LLM APIs are used — contact your Rasa account team. Rasa supports fully self-hosted model deployments including fine-tuned Llama models.
</Note>

## Deployment checklist

Use this checklist to confirm your deployment is production-ready before routing live traffic to it.

**Environment and licensing**

- [ ] `RASA_LICENSE` environment variable is set in your container or pod environment
- [ ] License is valid for your expected conversation volume (Developer Edition supports up to 1,000 conversations per month)

**Infrastructure**

- [ ] Tracker store configured — PostgreSQL is recommended for production (SQLite is the default and is not suitable for concurrent traffic)
- [ ] Event broker configured for analytics — use Kafka or RabbitMQ; required for the Studio analytics dashboard
- [ ] S3-compatible object storage configured for model files (required for multi-pod Kubernetes deployments)

**Agent configuration**

- [ ] `assistant_id` set in `config.yml` — this uniquely identifies your deployment in analytics and logs; each deployed agent should have a distinct value:
  ```yaml config.yml
  assistant_id: my-production-agent
  ```
- [ ] LLM provider API keys are set as environment variables (not hardcoded in config files)
- [ ] Action server endpoint is set in `endpoints.yml` and the action server is running
- [ ] All `action_*` steps referenced in flows have implementations in the action server

**Testing**

- [ ] End-to-end conversation tests pass against the deployed environment
- [ ] Filler messages verified in voice channels (if applicable)
- [ ] Fallback and error paths tested with unexpected user input

## Connect to Rasa Studio

If your team uses [Rasa Studio](https://rasa.com/docs/studio/intro/) for no-code flow authoring and analytics, point Studio at your deployed Rasa server after deployment. In Studio's connection settings, set the action endpoint to your Rasa server's public URL.

Once connected, your team can review conversations, update flows visually, and trigger retraining — all without direct access to the Rasa server.
