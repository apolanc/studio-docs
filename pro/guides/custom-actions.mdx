Custom actions are Python functions that run during a conversation whenever a flow reaches an `action` step. They let you go beyond static responses and connect your agent to the real world — querying a database for live account data, calling a payment API, applying business rules, or generating personalised messages. When a custom action finishes, it returns events that update the conversation state, and the flow continues from where it left off.

## How the action SDK works

Every custom action is a Python class that extends `Action` from the `rasa_sdk` package. You implement two methods:

- **`name()`** returns the action's string identifier — this must match the name used in `flows.yml` and `domain.yml`.
- **`run()`** contains your logic. It receives the current conversation state and returns a list of events.

The three objects passed into `run()` give you everything you need:

| Object | Purpose |
|---|---|
| `dispatcher` | Send messages directly to the user |
| `tracker` | Read conversation state (slots, messages, sender ID) |
| `domain` | Access domain configuration |

### Reading conversation state

Use the `tracker` to access anything the conversation knows:

```python
# Get the value of a named slot
amount = tracker.get_slot("amount")

# Get the full latest user message object
latest_message = tracker.latest_message

# Get the raw text of the latest user message
user_text = tracker.latest_message.get("text")

# Get the conversation ID
conversation_id = tracker.sender_id
```

### Returning events

Actions communicate back to the conversation by returning a list of events. The most common is `SlotSet`, which stores a value the flow can branch on:

```python
from rasa_sdk.events import SlotSet

return [SlotSet("has_sufficient_funds", True)]
```

## Example: check sufficient funds

This action reads the `amount` slot, checks it against a user's balance, and sets a `has_sufficient_funds` slot that the flow uses for branching.

```python
from typing import Any, Text, Dict, List
from rasa_sdk import Action, Tracker
from rasa_sdk.executor import CollectingDispatcher
from rasa_sdk.events import SlotSet

class ActionCheckSufficientFunds(Action):
    def name(self) -> Text:
        return "action_check_sufficient_funds"

    def run(
        self,
        dispatcher: CollectingDispatcher,
        tracker: Tracker,
        domain: Dict[Text, Any],
    ) -> List[Dict[Text, Any]]:
        # In production, retrieve the balance from your banking API
        balance = 1000
        transfer_amount = tracker.get_slot("amount")
        has_sufficient_funds = transfer_amount <= balance
        return [SlotSet("has_sufficient_funds", has_sufficient_funds)]
```

### Using the action in a flow

Add an `action` step to your `flows.yml` and use the slot it sets to branch:

```yaml
steps:
  - collect: amount
  - action: action_check_sufficient_funds
    next:
      - if: not slots.has_sufficient_funds
        then:
          - action: utter_insufficient_funds
            next: END
      - else: proceed
```

### Registering the action in domain.yml

Add the action name to the top-level `actions` list in `domain.yml` so Rasa can validate your project:

```yaml
actions:
  - action_check_sufficient_funds
```

## Running the action server

Rasa supports two modes for executing custom actions. Choose based on your environment.

<Tabs>
  <Tab title="Module mode (local dev)">
    In module mode, Rasa imports your actions directly — no separate process needed. This is the fastest setup for local testing.

    Uncomment the `actions_module` line in `endpoints.yml`:

    ```yaml
    action_endpoint:
      actions_module: actions
    ```

    The value `actions` refers to the `actions/` directory (or `actions.py` file) in your project root. Run your assistant normally:

    ```bash
    rasa inspect
    ```

    <Note>
      Module mode is convenient for development but is not recommended for
      production, as your action code runs inside the Rasa process itself.
    </Note>
  </Tab>
  <Tab title="Server mode (production)">
    In server mode, your actions run in a separate HTTP service that Rasa calls over the network. This is the recommended approach for staging and production.

    Start the action server:

    ```bash
    rasa run actions
    ```

    This starts the action server on port `5055` by default. Point `endpoints.yml` at it:

    ```yaml
    action_endpoint:
      url: http://localhost:5055/webhook
    ```

    In a containerised deployment, replace `localhost` with your action server's service name or IP address.
  </Tab>
</Tabs>

## Error handling

Custom actions can fail — APIs go down, databases return unexpected results, slots hold unexpected types. Always wrap your logic in a `try/except` block and send a user-friendly fallback message through `dispatcher`:

```python
from typing import Any, Text, Dict, List
from rasa_sdk import Action, Tracker
from rasa_sdk.executor import CollectingDispatcher
from rasa_sdk.events import SlotSet

class ActionFetchAccountBalance(Action):
    def name(self) -> Text:
        return "action_fetch_account_balance"

    def run(
        self,
        dispatcher: CollectingDispatcher,
        tracker: Tracker,
        domain: Dict[Text, Any],
    ) -> List[Dict[Text, Any]]:
        try:
            # Replace with a real API call in production
            balance = fetch_balance_from_api(tracker.sender_id)
            return [SlotSet("account_balance", balance)]
        except Exception as e:
            dispatcher.utter_message(
                text="Sorry, I couldn't retrieve your balance right now. Please try again in a moment."
            )
            return []
```

<Tip>
  Prefer defining error and fallback messages as named `utter_*` responses in
  `domain.yml` and triggering them via flow branching, rather than calling
  `dispatcher.utter_message()` with inline text. Named responses are easier to
  validate, translate, and A/B test.
</Tip>
