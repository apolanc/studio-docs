A Maestro agent is defined by three files working together: `flows.yml` describes the tasks your agent can handle, `domain.yml` declares the slots and responses it uses, and `config.yml` configures the underlying LLM pipeline. Maestro reads all three at runtime to understand what the agent can do, when to do it, and how to respond.

## The anatomy of an agent definition

```text
your-project/
├── config.yml          ← pipeline components, LLM, policies
├── domain.yml          ← responses, slots, actions
└── data/
    └── flows.yml       ← task flows with descriptions and steps
```

Each file has a distinct responsibility:

| File | What it controls |
|---|---|
| `flows.yml` | The tasks your agent handles — description-driven, slot-based |
| `domain.yml` | Responses the agent can send, slots it can fill, and custom actions it can call |
| `config.yml` | Which LLM to use, which command generator, and which policies to apply |

## Defining flows

Flows are the building blocks of agent behaviour. Each flow represents a discrete task your agent can complete — checking an account balance, submitting a support ticket, processing a refund. Maestro activates a flow when the user's input matches its description.

```yaml data/flows.yml
flows:
  handle_billing:
    description: Handle billing inquiries, account charges, and invoice requests.
    steps:
      - collect: billing_issue
        description: What billing issue does the user have?
      - action: action_fetch_invoice
      - action: utter_billing_resolved
```

### Why descriptions matter

The `description` field on a flow is what Maestro reads to decide when to activate it. When a user sends a message, the command generator compares their input against every flow description to determine which flow to start. A vague or overlapping description leads to poor routing; a clear, specific description leads to accurate dispatch.

Write descriptions in plain language that mirrors what your users actually say:

| Less effective | More effective |
|---|---|
| `Handle billing.` | `Handle billing inquiries, account charges, and invoice requests.` |
| `Account stuff.` | `Answer questions about account details, profile settings, and login issues.` |

Individual `collect` steps can also have descriptions. Use them when the slot name alone is ambiguous:

```yaml data/flows.yml
- collect: amount
  description: The number of US dollars to transfer.
```

### Flow step types

You have several building blocks available inside a flow:

- **`collect`** — ask the user for a value and store it in a slot
- **`action`** — run a response (`utter_*`) or a custom action (`action_*`)
- **`set_slots`** — assign slot values directly without prompting the user
- **`link`** — hand off to another flow when this one completes
- **`call`** — invoke a sub-flow mid-conversation and return here when it finishes
- **`next` with `if`/`else`** — branch based on slot values or conditions

A complete flow with branching looks like this:

```yaml data/flows.yml
flows:
  handle_billing:
    description: Handle billing inquiries, account charges, and invoice requests.
    steps:
      - collect: billing_issue
        description: What billing issue does the user have?
      - action: action_fetch_invoice
        next:
          - if: slots.invoice_found
            then: billing_resolved
          - else:
              - action: utter_invoice_not_found
                next: END
      - action: utter_billing_resolved
        id: billing_resolved
```

## Defining slots in the domain

Every slot your flows reference must be declared in `domain.yml`. The `from_llm` mapping type tells Rasa to extract the slot value from the user's message using the active language model — this is the standard approach for open-ended text slots.

```yaml domain.yml
slots:
  billing_issue:
    type: text
    mappings:
      - type: from_llm

  invoice_found:
    type: bool
    mappings:
      - type: controlled
```

Use `type: controlled` for slots that your custom actions set programmatically (not extracted from user input). Use `type: from_llm` for anything the user provides in natural language.

## Defining responses

Responses live in the `responses` section of `domain.yml`. Any step that calls `utter_*` must have a matching entry here:

```yaml domain.yml
responses:
  utter_ask_billing_issue:
    - text: "What billing issue can I help you with today?"

  utter_billing_resolved:
    - text: "I've resolved your billing issue. Is there anything else I can help you with?"

  utter_invoice_not_found:
    - text: "I wasn't able to find an invoice for your account. Let me connect you with the billing team."
```

Rasa automatically generates the question prompt for a `collect` step by looking for a response named `utter_ask_{slot_name}`.

## Listing custom actions

Any `action_*` step that calls a Python function must be declared in the `actions` section of `domain.yml`:

```yaml domain.yml
actions:
  - action_fetch_invoice
```

Custom actions run in a separate action server process. See [Custom Actions](/pro/guides/custom-actions) for implementation details.

## Pipeline configuration

`config.yml` ties everything together by specifying the LLM-powered components:

```yaml config.yml
recipe: default.v1
language: en

pipeline:
  - name: CompactLLMCommandGenerator
    llm:
      model_name: gpt-4o

policies:
  - name: FlowPolicy
```

You can swap `gpt-4o` for any supported model. Rasa also supports fine-tuned smaller models (such as Llama 8B) for lower latency and predictable hosting costs.
