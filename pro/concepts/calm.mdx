CALM (Conversational AI with Language Models) is the dialogue system at the core of Rasa Pro. It powers every text and voice assistant you build on the platform, combining the natural-language flexibility of large language models with deterministic, auditable business logic. The result is an agent that understands anything a user might say — but only ever does what you've explicitly defined.

## The core insight: separate language from logic

Most LLM-based agent frameworks give the model full control: understand the user, decide what to do, generate the response — all in one pass. This is fast to prototype, but breaks down in production when business rules change, edge cases multiply, or a regulator asks why the agent said what it said.

CALM takes a different approach. The LLM's job is strictly **language understanding**: read the conversation and express what the user wants as a structured command. A separate, deterministic **dialogue manager** then decides what actually happens next by executing pre-defined flows. These two concerns never mix.

## How CALM processes a message

Every incoming user message travels through up to three layers before a response is sent.

### 1. Dialogue understanding

The LLM reads the full conversation history and the user's latest message together, then outputs a set of internal **commands** that describe how the conversation should progress. Commands are typed and structured — not free-form text. Examples include:

- `StartFlow("transfer_money")` — the user wants to begin a task
- `SetSlot(amount, 50)` — the user has provided a value
- `CancelFlow()` — the user wants to abandon what they started
- `StartFlow("pattern_chitchat")` — the user said something off-topic

Because the LLM outputs commands rather than prose, its reasoning is inspectable and its mistakes are reproducible. If the model misreads a message, you can see exactly which command it generated and why.

### 2. Dialogue management

The **FlowPolicy** receives the commands and executes them against your flow definitions. At this stage, the LLM is no longer involved — every decision is deterministic. The dialogue manager will:

- Start, pause, or resume flows based on what the commands say
- Advance through collect steps to gather missing information
- Evaluate branching conditions against current slot values
- Trigger actions — custom Python code, API calls, or built-in responses
- Hand off to [conversation pattern flows](/pro/concepts/conversation-patterns) when users go off-script

Because business logic lives in YAML flows rather than in a prompt, it's versioned in source control, testable, and easy to audit.

### 3. Contextual response rephraser (optional)

By default, your assistant sends the templated responses you define in `domain.yml`. If you want responses to sound more natural in context — especially when recovering from interruptions or corrections — you can enable the **Contextual Response Rephraser**. A second LLM pass rewrites the templated text to fit the flow of the specific conversation, without changing its meaning or adding unsanctioned content.

You can enable the rephraser globally or limit it to specific response types (for example, only pattern flows).

## Why this architecture matters

The separation of language from logic gives you properties that pure-LLM approaches can't easily provide.

**No hallucinations in business logic.** Flows are YAML files. The LLM cannot invent steps, skip required fields, or bypass branching conditions. If a user asks the agent to transfer money without providing an amount, the `collect` step asks again — every time.

**Full debuggability.** When something goes wrong, you can inspect the exact command the LLM generated, the flow step that was active, and the slot values at that point. There's no unstructured chain-of-thought to interpret.

**Cost efficiency.** Because language understanding is a narrow, well-defined task, it can run on smaller fine-tuned models — including self-hosted Llama 8B — rather than the largest frontier models. Dialogue management adds zero inference cost.

**Scales with complexity.** Adding a new capability means adding a new flow, not rewriting a monolithic prompt. CALM's routing is designed to handle dozens of flows without degrading accuracy.

### Command examples

The following commands are typical outputs from the dialogue understanding step. They drive all routing and slot-filling decisions — no free-form text is passed to the dialogue manager.

```
StartFlow("transfer_money")       # user wants to start a task
SetSlot(amount, 50)               # user provided a value
CancelFlow()                      # user wants to abandon the current flow
StartFlow("pattern_chitchat")     # user said something off-topic
```

## CALM vs. other approaches

The table below compares CALM to two common alternatives: ReAct-style LLM agents that use tool-calling in a loop, and classic intent-based NLU bots.

| | LLM-Centric (ReAct) | CALM (Rasa) | Classic NLU |
|---|---|---|---|
| **Business logic** | Embedded in prompts — inconsistent, hard to version | Deterministic YAML flows — versioned and auditable | Hardcoded dialogue trees |
| **Scalability** | Degrades as more tasks are added; cascading errors | Scales reliably — modular flows don't interfere | NLU accuracy drops with topic overlap |
| **Debuggability** | Hard — must read through unstructured text | Easy — discrete commands pinpoint every decision | Rigid but traceable |
| **Topic changes** | Often derails the agent | Built-in conversation repair via pattern flows | Typically breaks |
| **Production cost** | High — multiple LLM calls per turn | Efficient — fine-tuned models, one LLM call per turn | Very low — no LLM needed |

## Smaller models, same results

CALM works with GPT-4 and other frontier models out of the box. It's also designed to run on fine-tuned models as small as Llama 8B. Using a smaller self-hosted model means:

- **Lower latency** — critical for voice assistants where response time is perceptible
- **Predictable cost** — fixed hosting rather than per-token billing
- **Data sovereignty** — conversation data never leaves your infrastructure
